## 2026-07-21 品牌新名稱承接：事件理由／承接證據／使用者範圍三層判斷

- Case record: `ancuiyan-aesthetic-medicine-brand-personal`。使用者的個人避買理由是 ETtoday 對愛爾麗偷拍事件的報導；CTWANT 留存的 Facebook Page transparency 截圖則只用來辨識同一 Page 曾由「愛爾麗診所 台中+」改為「安萃妍醫美 新竹館」。
- 處理舊品牌事件後出現新名稱時，必須先明確拆開三層：**事件理由**（為何避開）、**名稱／帳號承接證據**（如何把舊事件對應到新名稱）、**使用者指定範圍**（單一據點、帳號或全品牌）。不可把名稱變更本身寫成避雷理由，也不可讓單一 Page 的證據自行升格為全體法人／分館同一性的外部事實。
- `personal` 記錄不以正式法人更名、商標或醫療登記證明作為使用者指定避買的前置門檻；但來源文字必須保留媒體歸屬與非終局法律口徑。若使用者明確指定全品牌／後續分館，應將範圍寫成使用者偏好，並清楚說明它不是外部證實全體實體同一。
- 執行前先用一句結構化回覆核對：`理由＝…；承接證據＝…；避雷範圍＝…`。若使用者已明確要求加入／列入，或後續明確要求完成入庫，才將有明確品牌名稱與既有事件／承接來源的案件路由為「品牌承接個人避雷」；敏感事件、媒體、法律程序與泛稱仍不得成為 searchable fields。
- 「你應該能理解我會不會把他加入避雷」這類反問是在測試意圖判斷，不單獨構成寫入授權；先直接回答推論或只釐清一題，不得因此自行建立 queue／寫入。後續若使用者再明確要求完成，才啟動正式入庫流程。

3. 補 `data/records/*.mjs`，跑 merge。
4. validator 正查、必要 `--expect` / `--not-expect`、exact generic field guard PASS。
5. 更新本檔必要口徑。
6. commit + push origin/main。
7. 不做 Pages / Pinecone，除非使用者要求。
